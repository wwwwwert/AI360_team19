## `AnomalyIntervalOverlapSearch`

Класс решает следующую задачу: **дан набор временных рядов, пользователь указывает один «запросный» ряд и временное окно, в котором он наблюдает аномалию. Нужно найти, в каких ещё рядах в это же временное окно тоже были аномалии.**

### Алгоритм работы

1. **Инициализация** — принимает объект `AnomalyDetectionSystem` (обёртку над детектором, например `ARDevDetector`), а также параметры поиска:
   - `max_gap` — максимальный допустимый промежуток между аномальными точками, чтобы они считались одним интервалом
   - `min_points` — минимальное число аномальных точек в интервале
   - `metric` — метрика перекрытия для ранжирования (`recall`, `precision`, `iou`, `f1`)
   - `metric_weights` — опциональные веса для взвешенной комбинации метрик
   - `overlap_threshold` — порог score, выше которого ряд считается «совпавшим»
   - `required_metrics` — дополнительные минимальные пороги на отдельные метрики

2. **Метод `search`** — основная точка входа:
   - Получает многомерный DataFrame, имя/индекс запросного ряда, временное окно `[start_time, end_time]` и опционально список кандидатов
   - **Детекция аномалий поштучно** (`_detect_each_series`): каждый столбец оборачивается в одномерный DataFrame и прогоняется через детектор **отдельно**, а не как многомерный ряд. Это принципиальный выбор — мультивариантный `ARDevDetector` возвращает общий агрегированный score, а здесь нужны **поточечные метки аномалий для каждого ряда отдельно**
   - **Извлечение интервалов** (`extract_anomaly_intervals`): бинарная маска `is_anomaly` в пределах окна превращается в список объектов `AnomalyInterval`. Соседние аномальные точки группируются, близкие группы сливаются (с учётом `max_gap`), короткие отбрасываются (`min_points`)
   - **Вычисление метрик перекрытия** (`compute_interval_overlap_metrics`): для каждого ряда-кандидата маска его аномальных интервалов сравнивается с маской запросного ряда побитово. Считаются:
     - **Recall** — доля аномальных точек запроса, покрытых кандидатом
     - **Precision** — доля аномальных точек кандидата, попавших в запрос
     - **IoU** (Intersection over Union) — пересечение / объединение
     - **F1** — гармоническое среднее precision и recall
   - **Скоринг и матчинг**: по выбранной метрике (или взвешенной комбинации) каждому кандидату присваивается score. Ряд считается «совпавшим» (`matched=True`), если score ≥ `overlap_threshold` и выполнены все `required_metrics`

3. **Результат** — объект `AnomalyIntervalSearchResult`, содержащий:
   - Все найденные интервалы по каждому ряду
   - Метрики перекрытия
   - Список совпавших рядов (`.matching_series`)
   - Метод `.to_frame()` для удобного просмотра в виде таблицы

### Вспомогательные структуры

- **`AnomalyInterval`** — замороженный dataclass, описывающий один непрерывный аномальный интервал (временные границы, индексы, число аномальных точек)
- **`IntervalOverlapMetrics`** — замороженный dataclass с четырьмя метриками перекрытия и счётчиками точек
- **`SeriesIntervalMatch`** — результат по одному ряду: интервалы, метрики, score, флаг совпадения

## `anomaly_interval_overlap_example.ipynb`

Ноутбук демонстрирует использование этого класса на практическом примере:

1. **Загрузка данных** — читается CSV с многомерным временным рядом (несколько метрик/сенсоров с общим временным индексом)

2. **Настройка детектора** — создаётся `AnomalyDetectionSystem` с `ARDevDetector` и подбираются параметры (порядок AR, порог, стабильный режим)

3. **Выбор запросного ряда и окна** — пользователь указывает конкретный ряд и временной диапазон, в котором визуально наблюдается аномалия

4. **Запуск поиска** — вызывается `AnomalyIntervalOverlapSearch.search()`, который:
   - Прогоняет детектор по каждому ряду независимо
   - Находит аномальные интервалы в указанном окне
   - Считает метрики перекрытия с запросным рядом

5. **Анализ результатов** — выводится таблица `.to_frame()` с рядами, отсортированными по score, и визуализация: графики рядов с подсвеченными аномальными интервалами, показывающие, какие ряды «аномалили» одновременно с запросным

По сути, ноутбук реализует сценарий **корреляционного анализа аномалий**: «если в ряде A произошла аномалия, в каких ещё рядах в это же время что-то пошло не так?» — что полезно для root cause analysis в мониторинге инфраструктуры и промышленных систем.
